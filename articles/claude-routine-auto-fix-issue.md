こんにちは、uji です。

ご自身の開発するソフトウェアで、バグや改善要望などが小さな Issue としてどんどん積み上がり、対処しきれなくなったご経験はありますか？
所属チームでメンテしている社内ツールでもご多分に漏れず、Issue が溜まって対処が滞りそうな風潮がありました。そこで、Claude Code の Routines 機能を使って、GitHub Issue を自動修正する仕組みを構築してみました。4人のメンバーで cron 設定を少しずらすだけで、**平日 15 分に 1 回 AI が Issue を拾いに行く状態を Extra Usage なしで実現**できています。

2週間ほど運用して、従来の類似手段（ローカル作業・GitHub Actions）と比較した良い点・悪い点が見えてきたのでまとめます。

:::message
今後、各種ツールのアップデートによって記事の説明と合わない部分がでてくる可能性がありますのでご注意ください。
:::

## Claude Code Routines とは？

Claude Code Routines は、Claude Code を定期実行できる機能です。
実行先は、ローカル PC と Anthropic 管理のクラウド上で動く環境（ローカル PC の電源を切っていても実行される）の 2 種類から選ぶことができます。元々クラウド環境のみのサポートだったのですが、いつの間にかローカルもサポートされていました。

トリガーは cron 形式のスケジュール・API・GitHub イベントの3種類を組み合わせて設定でき、GitHub・ファイルシステム・シェルコマンドに加えて MCP コネクター（Slack、Linear 等30以上のサービス）にもアクセス可能です。
プラン毎に1日あたりの実行上限が設定されており、超過分は Extra Usage として追加課金されます。

https://code.claude.com/docs/en/routines

[claude.ai/code/routines](https://claude.ai/code/routines) からスケジュールを設定できます。

## やっていること

### クラウド環境の Claude Code Routines に `/auto-fix-issue` を登録する

チームメンバーがそれぞれ自分のアカウントに以下を設定しています。

- **プロンプト**: `/auto-fix-issue`
- **スケジュール**: `0 * * * 1-5`（平日 1 時間おき）

Routines は1人あたり一定回数を超えた利用は Extra Usage の消費になるので、あまり高頻度に回すことはできないのですが、
4 人のメンバーでうまく設定している（0 の部分を 15,30,45 とずらしている）ので、Extra Usage を利用せずに **15 分に 1 回は誰かの Routines が Issue を拾いに行く状態**になっています。数は正義。

設定画面はこちら

![](https://static.zenn.studio/user-upload/c4253678ef9b-20260505.png)

「**プルリクエストの自動修正**」 を有効にするのが肝です。これを設定することで、 `/autofix-pr` のように PR 上のフィードバックに自動で対応してくれます。

![](https://static.zenn.studio/user-upload/93d2211fe701-20260505.png)

### `/auto-fix-issue` スキルの中身

Issue を選んで PR を作るまでのロジックはリポジトリの Skill にしています。おおまかな流れは以下

1. `gh issue list` でオープンな Issue を取得し、すでに PR がある Issue はスキップ
2. 対応できそうな Issue を 1 件選んで `claude/fix-<番号>` ブランチを作成
3. コードを最小限に修正してコミット・push
4. PR を作成
5. Issue にコメントを残す

スキルにロジックを切り出したことで、プロンプトの改善を全員に反映するとき、各自の Routines 設定を変えなくてよくなっています。`/auto-fix-issue` というプロンプトは変わらず、スキル本体を更新するだけで全員に同期される仕組みです。

### 機械的な処理は GitHub Actions に移譲する

PR が作成されると GitHub Actions 側でいくつかの処理が走ります。

- **レビュアー、アサイン、ラベル付け**
  - デフォルトで Issue の起票者をアサインに入れることで、起票者が PR を探しやすくなるようにしている
- **自動コードレビュー**
  - PR が open されると `/review-pr` スキルによる Claude のレビューが自動で入るようにしている
- **プレビュー環境のデプロイ**
  - フロントエンドのコードを変更する PR が open されると Cloud Run にプレビュー Revision が自動デプロイされる

ルールが決まっており、LLM にさせる必要性が薄い処理は Actions に任せ、Routines 側は複雑な仕事に集中できる分業になっています。

### フロントエンドの動作確認もローカル不要

PR を作成すると Cloud Run に `pr-<番号>` タグ付きのプレビュー Revision が自動デプロイされます。タグ付き URL でその PR のビルドにアクセスできるので、UI の変更もブラウザで確認が完結します。

1. Routines が PR を作って
2. Actions がプレビューをデプロイして
3. レビュアーがプレビュー URL で確認する

という流れが全てローカル環境なしで回ります。

## クラウド環境の Routines 利用で得られる恩恵

### コーディングエージェントに指示を投げる心理的ハードルが下がる

ローカルで Claude Code に指示を出すには「PC を開く」「作業中のブランチを退避する」といった暗黙の準備コストがあり、軽微な Issue ほど「後でやろう」と先送りになりがちでした。
Routines をクラウドで動かしている場合、スマートフォンから claude.ai を開いて Issue を立てるだけで動かせます。移動中・離席中・PC が手元にない場面でも「思いついたら投げる」ことができるので、Issue に対して行動するまでの閾値がかなり下がります。

また Routines で定期的に拾いに行く設定にしておけば、自分がアクションを取らなくても積み上がった Issue が勝手に処理され始めます。結果として、後回しにしていた小さな改善がチームとして自然と消化されるようになりました。

### サンドボックスで動いている安心感

クラウド環境で動作する Routines はローカルの開発環境を全く触りません。依存関係を壊したり、git のステートを汚したりする心配がないので、バックグラウンドで動いていることを気にせず手元の作業を続けられます。
ローカル環境で worktree で複数の作業を並行させているときも干渉しないのが地味にありがたいです。

Claude Code のクラウドサンドボックス環境は、リリース時は制約がきつくてあまり自由度がない印象でしたが、インストール済みツールが充実してきており、やれることも増えそうです。

https://code.claude.com/docs/ja/claude-code-on-the-web#%E3%82%A4%E3%83%B3%E3%82%B9%E3%83%88%E3%83%BC[…]%E6%B8%88%E3%81%BF%E3%83%84%E3%83%BC%E3%83%AB

:::message
一方で、クラウドサンドボックス環境では外部サービスとの認証情報（API キーや OAuth トークン）を安全に渡す方法が現時点でサポートされていません。Claude コネクタ以外の外部連携が必要な修正や、プライベートリポジトリの社内ライブラリのインポートには対応できないのが現状です。ローカル PC で Routines を動かす設定にすれば回避できます。
:::

### 質問せずに完遂まで持っていこうとしてくれる

「これはどう対応しますか？」といった割り込みをせずに完遂まで持っていこうとしてくれます。Routines はバックグラウンドで黙々とやり遂げてくれるので、コンテキストスイッチが一切発生しません。
気がついたら PR ができており、プレビューですぐに試せる、という体験は想像以上に快適でした。

大きな新機能の開発の場合、しっかりプランを練ってから実装に進んだ方が良い結果になることも多いですが、その場合でもプロトタイプの使用感を試せたり、実装に進んで初めて気付くことができる考慮事項の検知などにもつながり、ひとまず雑に投げてみるやり方はアリなのではないでしょうか。

ただし、要求が大きくなるほど精度は落ちます。クラウド環境だとリソース制約による失敗も起きることがあります。Issue や PR を細かく分割して指示できると成功率が上がります（最近発表された GitHub の [Stacked PRs](https://github.github.com/gh-stack/) は使い所だったりするのでしょうか…）。

### 自動起票の仕組みを拡充させることでメンテナーを本質的な業務に集中させられる

Issue の自動修正が機能するためには、そもそも修正対象の Issue が継続的に供給される必要があります。Routines と組み合わせると効果的なのが、「Issue 自動起票」の仕組みです。

例えば、我々のリポジトリには、毎朝 JST 6:00 に走る `daily-fix.yml` という GitHub Actions があります。前日マージされた PR のレビューコメントをスキャンして、スレッド単位でグループ化したうえで Claude Haiku が分類します。PR 作者が「後で直します」「次の PR で対応します」といった返信をしている指摘を未対応と判定し、重複チェックをしてから `daily-fix` ラベル付き Issue として自動起票します（対象ファイル・レビューコメント本文・diff も Issue に含まれます）。
これにより「レビューで先送りにした指摘」が自動的に Issue として蓄積され、Routines がそれを拾って修正するというループが回ります。人が意識して Issue を立てなくても Issue が供給される状態です。

他にも、Slack のリアクションから起票したり、Notion の AI Meeting 議事録から起票したり、とやれそうなことはいくらでも例が挙げられそうです。
こういった「Issue を自動で作る側の仕組み」を整えていくと、エンジニアは起票作業から解放されて本質的な業務に集中できます。

:::message
GitHub Issue がインターフェースとして適切かどうか、は議論の余地がありそうですが、Claude を始め多くのコーディングエージェントは特に指示をせずとも gh コマンドを使ってよしなに参照・操作してくれるので、AIファーストなインターフェースの一つにはなっているのではと思います。
:::

### トレードオフ：セッションをチームで共有できない

各自のアカウントで Routines が独立して動いているため、cron 式や権限周りなど Skill の外の部分の設定を変えたくなった際、変更作業が個人に依存してしまいます。チームで仕組みを共有する際のセットアップがもう少し楽になると嬉しいですね。
また、誰がどの Issue を取りに行くかで競合が起きる可能性があります。「既に PR がある Issue はスキップする」ロジックでほぼカバーできていますが、ほぼ同時に走った場合のレースコンディションは完全には防げていません。今回のユースケースだと、競合が起きても2~3ほどのPRが作られるだけなので大きな問題にはなっていません。（むしろ、より良い実装を選択し、採用できる副次的効果がありました。）
加えて、どのメンバーのセッションで実行されるかによって PR の作成者がそのメンバーのアカウントになる点も Nits なデメリットとして挙げられます。Auto review の通知もPR作成者に届くため、一見誰の担当PRかわかりづらくなることがあります。assignee やレビュアーを明示的に指定することである程度解消できます。

Routines は新しい機能なので、今後のアップデートに期待したいところです。

## Claude Code GitHub Actions と比較したメリット

人にこの仕組みを共有する際、よく聞かれるのが「Claude Code GitHub Actions との違いは？」でした。

### レビューコメントや CI の結果を見て修正してくれる

これが個人的に一番大きいと感じています。

Actions ベースの自動修正は「PR を作ったら終わり」になりがちです。CI が落ちていても次のアクションがありません。Routines は 「**プルリクエストの自動修正**」 を有効にしておくと、CI の実行結果を確認して、失敗していたら追加でコミットを積んで修正してくれます。「PR を作って CI が通るまで面倒を見る」というループを自動で閉じられるのが強みです。
Claude Code Action での自動レビューに対する返信や修正も上手くやってくれています。

### プロンプトがコードから比較的近い場所に残りやすい

ローカルで Claude Code に指示した場合、プロンプトはターミナル履歴やローカルセッションの中で消えてしまいます。「なぜこのコードが入ったのか」の背景が、コードや Git 履歴に残りません。
Routines 経由だと「Issue → PR → コミット」という GitHub 上の導線に修正の意図が刻まれます。Issue 本文がそのまま AI への指示書になっており、後から `git blame` → PR → Issue を辿れば文脈をたどれます。将来コードを読む人が PR から「このコミットが入った経緯」を追跡しやすくなります。
AI が生成するコードの比率が増えていくほど、意図のトレーサビリティは価値を持ちます。構造として残しやすい Routines の特性は、じわじわ効いてくると感じています。

### GitHub Actions 環境の制約とセットアップコスト

GitHub Actions で Claude を動かす場合、ランナー環境のセットアップ（ツールのインストール、`ANTHROPIC_API_KEY` などの認証情報のセットアップ、GitHub への push 権限の付与）をワークフロー側で組む必要があります。公式の `anthropics/claude-code-action` を使う場合でも、Actions のランナー環境制約は受けます。

クラウド Routines は Anthropic 管理のサンドボックスで Claude Code とよく使うツールがプリインストールされた状態で動き、claude.ai のアカウント連携で GitHub への認証も完結するので、ワークフロー側の準備はほぼ不要です。

また、LLMの利用料金がサブスクプランの料金から外れてAPI利用の従量課金になることに加えて、GitHub Actions 実行時間に応じて料金が発生するのもポイントです。PR の自動修正が積み重なると無視できない金額になる可能性もあるかもしれません。

一方で Actions の強みは、push / PR open / issue labeled などの GitHub イベントをより柔軟にトリガーにできる点です。PR コメントへの即応や push 直後のフォーマット適用など、リアルタイム性が求められる処理は Actions のほうが素直に書けます。

## おわりに

Issue の自動対応は Routines の活用の一例に過ぎず、「定時に勝手に動いてほしい繰り返し作業」は Routines 活用の余地がありそうです。
例えば、新しいライブラリの変更をキャッチアップして週次で Slack に要約を投げる、週次でバンドルサイズのトレンドを確認して劣化を検知する、といった定常的な保守作業も得意分野です。

Codex Automations などの同種の自動化サービスでも近い体験は得られそうですが、まだ十分に試せていないため比較は割愛します。チームの状況や既存のエコシステムに合わせて検討してみてください。

皆さんはどんな繰り返し作業を Routines で自動化していますか？セッション共有の回避策や Codex Automations との比較、より良い Issue 起票の仕組みなど、知見をお持ちの方はぜひコメントで共有いただけると嬉しいです。
